# 誕生日当てゲーム（二分探索）～ オープンスクール探究授業用 ～

中学生がプログラムを操作し、高校生（アシスタント役）の誕生日を
**二分探索（にぶんたんさく）**で当てるデモ教材です。

## ねらい

- 「Yes/No の質問を繰り返すと、少ない回数で答えにたどり着く」という
  二分探索のアルゴリズムを、実際に手を動かして体感する
- 配られたプログラムには **1ヶ所だけバグ** があり、それを見つけて直すと
  正しく動くようになる「デバッグ体験」をする
- 1ヶ所の修正が、月（1〜12）にも日（1〜31）にも同時に効くことで、
  「同じロジックが繰り返し使える」というプログラミングの強さを実感する

## 進行イメージ（中学生 × 高校生アシスタント）

1. 高校生が「誕生日は心の中で決めておいてね」と伝える（言わない）
2. 中学生が画面の「▶ スタート」を押す
3. 画面に「誕生月は 6 以下ですか？」のような質問が出る
4. 中学生が高校生に口頭で質問し、高校生が「はい/いいえ」を答える
5. 中学生がその答えに合わせて画面の「✅ はい」「❌ いいえ」ボタンを押す
6. 月が確定 → 続けて日も同じ流れで質問
7. **最初はバグがあるため、8回（月）/ 10回（日）質問しても終わらない**
8. 「あれ、おかしいね」となったら、左のプログラムを一緒に見て
   `narrow_range` 関数の中の `low = mid` を `low = mid + 1` に直す
9. 「▶ スタート」でやり直すと、月は最大4回、日は最大5回でぴったり当てられる

## バグの内容（指導者向け）

`narrow_range` 関数の中、「答えが mid より大きい」と分かった場合の処理が

```python
low = mid       # ← バグ：mid をまだ範囲に残してしまう
```

になっています。正しくは

```python
low = mid + 1   # ← 修正：mid はもう違うと分かったので外す
```

です。範囲の幅が「2」のまま動かなくなるため、画面には
「⚠️ 8回質問しても、まだ月が [1 か 2] のどちらかに絞れません」のように表示され、
中学生にも分かりやすく「止まっている」ことが伝わります。

検証のポイント：1ヶ所を直すだけで、月（1〜12, 全パターン）も
日（1〜31, 全パターン）も、両方ともきちんと最大回数内で当てられることを
確認済みです（月：最大4回、日：最大5回）。

## ファイル構成

| ファイル | 生徒が触る？ | 役割 |
|---|---|---|
| index.html / style.css / engine.js / lib/ | ✗ | 画面と仕組み |
| **quiz.py** | **✓**（左パネルに表示されているのと同じ内容） | 二分探索のロジック本体 |

> quiz.py は配布・印刷用に同じ内容を別ファイルでも用意しています。
> 画面の左側に直接コードが表示されるので、生徒はファイルを開く必要はありません
> （プログラムを直接読みながら、その場で議論できる設計です）。

## 動かし方

`index.html` をダブルクリックでブラウザに開くだけです。
インターネット接続は不要です（Brythonを同梱しています）。

## GitHub Pages で公開する場合の注意

- リポジトリのルートに `index.html` を含むこのフォルダの中身をそのまま置いてください。
- **`.gitignore` に注意**：もしリポジトリ作成時に GitHub の
  「.gitignore テンプレート：Python」を選んでいる場合、そのテンプレートには
  `lib/` という行が標準で含まれており、`lib` という名前のフォルダが
  まるごと Git に追加されない（＝GitHub Pages上で404になる）ことがあります。
  このプロジェクトでは Brython 本体を `lib/` ではなく **`brython-lib/`** という
  フォルダ名にして、この問題を避けています。
- 同梱の `.gitignore` と `.nojekyll` ファイルも一緒にアップロードしてください
  （`.nojekyll` は、GitHub Pages が独自の処理（Jekyll）を行わず、
  ファイルをそのまま配信するようにするためのものです）。
- アップロード後、ブラウザの開発者ツール（F12）の Console タブに
  赤いエラーが出ていないか確認すると、トラブルの早期発見に役立ちます。
  `404` と表示された場合は、ファイルパスやファイル名（大文字・小文字を含む）が
  リポジトリ内の実際のファイルと完全に一致しているか確認してください。
  GitHub Pages は Windows/Mac と異なり、大文字・小文字を区別します。


## 当日の進め方のヒント

- 最初の1回はあえて「バグ版」のまま試してもらい、
  「あれ、終わらないね？なんでだろう？」と問いかけると効果的です
- 左のコードパネルを見せながら、`mid`（真ん中の数字）が
  「もう違うと分かったのに範囲に残っている」ことを一緒に確認すると、
  二分探索の「半分に分ける」という考え方が腑に落ちやすいです
- 月を直したら、日のほうも自動的に直っていることに気づかせると、
  「同じ関数を月にも日にも使っている」という再利用性の話につなげられます
